# 任务规划

把设计文档转化为详细的实施计划。假定执行者没有项目上下文、需要每一步都被明确告知：碰哪些文件、做什么操作、怎么验证结果。计划就是 bite-size 任务的完整拼图。DRY、YAGNI、频繁保存。

<HARD-GATE>
1. 必须基于已确认的设计文档或需求来创建计划，禁止凭空拆任务。
2. 计划中禁止使用"TODO""待定""后续补充"等占位符，每步必须有实际可执行内容。
3. 计划完成后必须对照设计文档做覆盖检查，确认无遗漏后才交付。
</HARD-GATE>

**开始时声明：** "我用 writing-plans skill 来做实施计划。"

## 能力边界

### 本技能负责
- 基于已确认的设计文档或需求，拆解为 bite-size 可执行任务
- 每个任务包含明确文件路径、操作步骤和验证方式
- 作为设计到执行的桥梁

### 本技能不负责
- 设计文档的编写 → 使用 **requirement-briefing**
- 任务的具体执行 → 使用 **subagent-driven-development** 或手动执行
- 任务完成后的验证 → 使用 **verification-before-completion**
- 无设计文档时的凭空拆任务

## Gotchas

1. **任务粒度太粗** — 一个任务超过 5 分钟就要拆。Agent 常把多个操作合并成一个任务，导致执行困难。
2. **计划中使用"TODO""待定"等占位符** — 每一步必须有实际可执行内容。Agent 常在计划中使用模糊表述，导致执行时无从下手。
3. **任务之间高度耦合但没有标注依赖** — 必须标注任务间依赖关系。Agent 常假设执行者会按顺序执行，但实际可能需要并行或回退。
4. **只给步骤不给验证方式** — 每个任务必须有验证方式。Agent 常忘记写"怎么确认这一步做对了"。
5. **计划没有基于设计文档对照检查** — 写完必须自查覆盖度。Agent 常遗漏设计文档中的某些要求。
6. **执行交接时不检查 subagent-driven-development 是否已安装** — Agent 直接将"子 Agent 编排执行"作为推荐选项呈现给用户，但该技能可能未安装，用户选择后才发现不可用。**纠正：在执行交接前检查 subagent-driven-development 是否可用，不可用则从选项中移除或标注"未安装"。**

## 范围检查

如果设计文档涵盖多个独立子系统，应该在需求梳理阶段就已经拆开了。如果没拆，先建议拆成多个独立计划。

## 文件结构

在定义任务之前，先理清要创建或修改哪些文件，每个文件负责什么。

- 每个文件有单一清晰的职责
- 一起改的文件放在一起。按职责拆，不按技术层拆
- 已有项目中沿用现有模式

## 任务粒度

**每一步是一个动作（2-5 分钟）：**
- "创建目录结构" — 一步
- "写测试用例" — 一步
- "跑测试确认失败" — 一步
- "写最小实现" — 一步
- "跑测试确认通过" — 一步
- "提交" — 一步

## 计划文档头部

每个计划必须以这段开头：

```markdown
# [任务名称] 实施计划

> **执行须知：** 使用 subagent-driven-development（推荐）来逐任务执行。步骤使用 `- [ ]` 勾选语法跟踪进度。

**目标：** [一句话描述要产出什么]

**方案概述：** [2-3 句方法说明]

**涉及技术/工具：** [关键工具、框架、依赖]

---
```

## 任务结构

```markdown
### 任务 N：[任务名称]

**涉及文件：**
- 创建：`exact/path/to/file.ext`
- 修改：`exact/path/to/existing.ext`
- 测试：`exact/path/to/test.ext`

- [ ] **步骤 1：[动作描述]**
  具体操作内容，包括确切的命令、参数、预期输出

- [ ] **步骤 2：[动作描述]**
  具体操作内容

- [ ] **步骤 3：验证**
  如何确认这一步做对了
```

## 禁止占位符

每一步必须有实际可执行的内容。以下都是计划失败：
- "TODO""待定""后续补充"
- "根据需要添加错误处理""补充验证逻辑"
- "参考任务 N"（把内容完整写出来，执行者可能乱序读任务）
- 只有描述没有具体操作的步骤
- 引用了前面任务没定义的东西

## 自查

写完计划后，对照设计文档做自查：

1. **覆盖检查** — 设计文档的每个要求都能指到一个任务吗？
2. **占位符扫描** — 搜索计划中的红旗词：TODO、待定、后续、参考任务N
3. **一致性检查** — 后面任务引用的文件名、变量名，跟前面任务定义的一致吗？

自查通过后，调用 EnterSpecMode 进入 spec 模式，展示最终计划。用户审核后调用 ExitSpecMode 确认，才进入执行交接阶段。

发现问题就地修掉。

## 执行交接

计划完成后，先检查 `subagent-driven-development` 技能是否在当前环境中可用（查看 `<available_skills>` 列表或检查 `%USERPROFILE%\.qoder-cn\skills\`（Windows）/ `~/.qoder-cn/skills/`（Unix）目录）。

然后使用 AskUserQuestion 工具询问用户执行方式：

**如果 subagent-driven-development 可用：**
- 选项 1（推荐）："子 Agent 编排执行" — 使用 subagent-driven-development 逐任务执行
- 选项 2："串行执行" — 逐个任务手动执行
- 选项 3："我自己执行" — 用户自行执行

**如果 subagent-driven-development 不可用：**
- 选项 1（推荐）："串行执行" — 逐个任务手动执行
- 选项 2："我自己执行" — 用户自行执行
- 选项 3："先安装 subagent-driven-development" — 暂停任务，提供安装命令：`git clone --branch subagent-driven-development/v0.1.0 https://epfa-gitlab.csvw.com/ecc-2/qoder-skills.git`

用户选择安装依赖时，提供安装命令并等待用户确认安装完成。

## 执行后复盘（自迭代钩子）

每次完成本 skill 的全部步骤后，Agent 必须自动执行以下动作，不询问用户：

1. **反思**：本轮执行中，是否遇到了正文和 Gotchas 都没覆盖的坑？触发条件：
   - Gotchas 中没写但实际踩了的计划问题
   - 某种任务类型的拆解粒度不当
   - 设计文档中某个需求被遗漏

2. **记录**：有则追加到 `evals/PITFALLS_LOG.md`，无则跳过。格式：

| 日期 | 场景摘要 | 犯了什么错误 | 如果早知道__就不会犯 | 临时补救方式 |
|------|---------|-------------|-------------------|------------|
| YYYY-MM-DD | 一句话 | Agent 具体做了什么 | 缺失的知识/规则 | 怎么绕过去的 |

3. **不提交**：`PITFALLS_LOG.md` 是个人本地文件，不提交到 skill registry。积累 >= 3 条同类型坑后，由维护者提炼为正式 Gotchas，更新 SKILL.md 并走迭代管线。
