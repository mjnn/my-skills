# 需求梳理

把模糊想法变成清晰方案，通过自然对话逐步细化需求。先理解背景，再一个问题一个问题地问，最后出设计等你确认。

<HARD-GATE>
在呈现设计并获得用户明确批准之前，禁止执行任何实施动作（写代码、创建文件、跑分析、出报告等）。这个规则适用于所有任务，无论看起来多简单。
</HARD-GATE>

## 能力边界

### 本技能负责
- 复杂任务前的需求梳理、背景理解和目标对齐
- 将模糊想法转化为清晰方案
- 设计文档的编写和用户确认

### 本技能不负责
- 任务的具体执行 → 由执行型 skill 负责
- 将设计拆成可执行任务 → 使用 **writing-plans**
- 任务完成后的验证 → 使用 **verification-before-completion**
- 已有明确设计文档时的重复梳理

## Gotchas

1. **"这个太简单了不需要设计"** — 简单任务最容易被未验证的假设坑。设计可以很短（几句话也行），但必须呈现并获得批准。Agent 常犯的错误是跳过设计直接写代码，最后返工。
2. **一次抛出多个问题** — 用户认知负荷高，一次只问一个问题。Agent 常犯的错误是列出 5 个问题让用户"选一个"，导致用户跳过或忽略关键问题。
3. **设计文档写成代码实现** — 设计是方案层面（做什么、怎么做），不是逐行代码。Agent 常在设计阶段就写出完整代码，导致设计文档失去"方案"意义。
4. **用户确认设计后忘记调用 writing-plans** — 设计→计划→执行是分离的三个阶段。Agent 常在设计确认后直接跳到执行，跳过计划拆解。
5. **只在对话中呈现设计，没有保存到文件** — 对话会丢失，必须持久化到项目文档目录。Agent 常犯的是只在对话中展示设计，不写入文件。
6. **不检查 writing-plans 是否已安装就直接执行** — Agent 到流程第 9 步才发现 writing-plans 不可用，用户体验断裂。**纠正：在流程开始前执行前置依赖检查，提前发现并让用户决策。**

## 防偷懒规则

"这个太简单了不需要设计"是最常见的坑。简单的任务恰恰是未经验证的假设浪费最多时间的地方。设计可以很短（简单任务几句话也行），但必须呈现并获得批准。

## 流程



### 运行时说明（无脚本）

本 skill 以对话与流程引导为主，不直接执行脚本。若会话中需要运行**本 skill 或其他 skill 的 scripts/**，必须先确认 skill-kit 已就绪（Read skill-kit `references/preflight.md`）；未就绪则引导用户运行 **skill-env-setup**。

你必须按顺序完成以下步骤：

0. **前置依赖检查** — 检查 `writing-plans` 技能是否在当前环境中可用（查看 `<available_skills>` 列表或检查 `%USERPROFILE%\.qoder-cn\skills\` 目录）。如果不可用，使用 AskUserQuestion 工具提示用户：
   - 问题："本技能在设计确认后需要调用 writing-plans 进行任务拆解，但当前环境未安装该技能。是否继续执行？"
   - 选项1："继续执行" — 在没有 writing-plans 的情况下继续，设计确认后手动完成任务拆解
   - 选项2："先安装依赖" — 暂停任务，提供安装命令：`git clone --branch writing-plans/v0.1.0 https://epfa-gitlab.csvw.com/ecc-2/qoder-skills.git`
   - 选项3："取消任务" — 终止当前任务
   - 用户选择"继续执行" → 记录限制，在过渡阶段（步骤 9）提示用户手动拆解任务
   - 用户选择"先安装依赖" → 提供安装命令，等待用户安装完成后再继续
   - 用户选择"取消任务" → 终止执行

1. **探索上下文** — 查看当前项目文件结构、相关文档、近期提交
2. **评估范围** — 如果需求涉及多个独立子系统，先帮用户拆成子项目，聚焦第一个
3. **逐一提问澄清** — 一次只问一个问题，理解目的、约束、成功标准。使用 AskUserQuestion 工具，每次提供 2-4 个选项，禁止纯文本提问。
4. **提出 2-3 个方案** — 各有取舍，给出你的推荐和理由
5. **分段呈现设计** — 按复杂度分段，每段等用户确认。信息收集完成后，调用 EnterSpecMode 进入 spec 模式，生成完整设计方案。用户审核后调用 ExitSpecMode 确认，才进入下一步。
6. **写出设计文档** — 保存到项目文档目录
7. **设计自查** — 检查遗漏、矛盾、模糊点
8. **用户审阅设计文档** — 请用户过目后再继续
9. **过渡到任务规划** — 如果步骤 0 确认 writing-plans 可用，调用 writing-plans skill。如果用户在步骤 0 选择"继续执行"（writing-plans 不可用），提示用户："writing-plans 未安装，请手动将设计文档拆解为可执行任务，或安装 writing-plans 后再执行。"

## 关键原则

- **一次一个问题** — 不要一下抛出好几个问题压人。使用 AskUserQuestion 工具分轮提问，每轮最多 4 个问题。
- **优先选择题** — 比开放式问题更容易回答
- **无情砍需求（YAGNI）** — 去掉所有不必要的部分
- **总是给多个方案** — 至少 2-3 个，有取舍有推荐
- **增量确认** — 设计分段呈现，每段确认后再继续
- **灵活回退** — 发现理解有偏差就回去澄清

## 设计隔离与清晰

- 把系统拆成独立的小单元，每个单元有明确职责、接口
- 对每个单元能回答：做什么、怎么用、依赖什么
- 小文件比大文件好

## 已有项目中的工作方式

- 先摸清现有结构再提改动
- 沿用已有模式；如果现有结构有问题影响当前任务，把改进纳入设计
- 不提无关的重构，只做服务于当前目标的事

## 设计文档

保存设计文档后做自查：
1. **遗漏检查** — 有没有"TODO""待定"或不完整的部分？
2. **一致性检查** — 各部分之间有没有矛盾？
3. **范围检查** — 是否聚焦到可以一次规划执行？
4. **模糊检查** — 有没有哪句话可以有两种理解？

修完问题后请用户审阅。

## 过渡

设计文档经用户确认后，如果 writing-plans 已安装，调用 writing-plans skill 创建实施计划。如果 writing-plans 未安装且用户在步骤 0 选择了"继续执行"，提示用户手动拆解任务。不要直接跳到执行。

## 执行后复盘（自迭代钩子）

每次完成本 skill 的全部步骤后，Agent 必须自动执行以下动作，不询问用户：

1. **反思**：本轮执行中，是否遇到了正文和 Gotchas 都没覆盖的坑？触发条件：
   - Gotchas 中没写但实际踩了的错误
   - 某个需求场景的澄清方向遗漏
   - 设计文档的自查项不够全面

2. **记录**：有则追加到 `evals/PITFALLS_LOG.md`，无则跳过。格式：

| 日期 | 场景摘要 | 犯了什么错误 | 如果早知道__就不会犯 | 临时补救方式 |
|------|---------|-------------|-------------------|------------|
| YYYY-MM-DD | 一句话 | Agent 具体做了什么 | 缺失的知识/规则 | 怎么绕过去的 |

3. **不提交**：`PITFALLS_LOG.md` 是个人本地文件，不提交到 skill registry。积累 >= 3 条同类型坑后，由维护者提炼为正式 Gotchas，更新 SKILL.md 并走迭代管线。
