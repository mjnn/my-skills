# 上汽大众结构化PPT生成器

基于内置标准模板，通过对话式引导完成信息收集、文案确认、排版设计、图表绘制，最终生成统一布局的PPT。

## 核心设计思想

> **不是"你给我个主题我帮你生成"，而是"你告诉我这个故事怎么讲，我帮你装进模板里"。**

- 模板是skill的静态资源（`templates/模板.pptx`），所有生成任务基于统一模板
- 所有提问必须通过 AskUserQuestion 工具进行，提供结构化选项，禁止纯文本提问
- 信息收集完成后必须进入 Spec 模式，用户审核计划后才能开始执行
- AI适度发挥，但用户说"自由发挥"时要引导或给选项，不能完全自由发挥
- 每页PPT从模板中替换元素和文案生成，保证可控性

<HARD-GATE>
在生成PPT之前，必须完成以下步骤：
1. 使用 AskUserQuestion 工具完成信息收集（禁止用纯文本提问）
2. 调用 EnterSpecMode 进入 spec 模式，生成执行计划
3. 用户通过 ExitSpecMode 确认计划后，才能开始生成

必须确认以下信息已完整提供：
1. 汇报主题
2. 汇报对象和场景
3. 叙事结构（章节规划）
4. 每页核心观点和支撑要点
5. 输出文件路径

如果以上信息缺失或未走 spec 模式，禁止生成PPT。
</HARD-GATE>

## 能力边界

### 本技能负责
- 基于内置标准模板生成"总分总、由浅入深"结构化叙事 PPT
- 通过对话式引导完成信息收集、文案确认、排版设计、图表绘制
- 生成统一布局的 PPT（封面→目录→正文→结尾）

### 本技能不负责
- 修改现有 PPT 的局部内容 → 直接编辑 PPTX 文件
- 非结构化叙事的创意展示 → 需要手动排版设计
- 数据分析和图表生成 → 由数据分析 skill 负责
- 需要手动排版设计的艺术创作

## Gotchas

**1. 图表位置过低遮挡页脚**
Agent 在生成PPT时忽略页脚占位符位置，导致内容遮挡页脚横线。纠正方案：必须先分析模板中的页脚位置，设定安全区域（内容 bottom < 6.4in），所有页面生成函数强制遵守。

**2. 过度追求炫酷效果而偏离模板规范**
Agent 尝试加入毛玻璃、渐变色、阴影等非模板元素。纠正方案：严格使用模板内置版式（封面、目录、标题、正文、结尾），禁止额外添加非模板内的视觉元素。

**3. 生成多个主题模板而非统一布局**
Agent 按不同汇报主题生成多个独立模板。纠正方案：建立单一的通用结构化叙事框架（封面→目录→【总】→【浅】→【中】→【深】→【总】），所有主题共用同一套布局函数和内置模板。

**4. 页面底部添加多余的导航标签条**
Agent 在每页左下角添加"01 总览""02 背景"等导航标签条，与页面标题重复，造成信息冗余。纠正方案：禁止在页面底部添加导航标签条，页面标题已经足够标识当前内容，不要画蛇添足。

**5. 硬编码品牌规范而非从模板提取**
Agent 直接在代码中写死颜色和字体。纠正方案：模板作为静态资源内置，品牌规范从模板提取后作为参数传入所有页面生成函数。

**6. 用户说"自由发挥"时真的自由发挥**
Agent 在用户没有明确指示时过度发挥。纠正方案：用户说"你看着办"或"自由发挥"时，必须给出选项让用户选择，不能直接自由发挥。

**7. 图表被拉伸变形（先生成图表后适配网格）**
Agent 先按默认尺寸生成图表图片，再强制塞入网格区域，导致图表宽高比严重变形。纠正方案：必须先在12×6网格中规划图表占位区域，计算实际宽高（英寸），再按该宽高比生成图表。流程是：网格规划→计算尺寸→生成图表→插入PPT，不能反过来。

**8. 多图页面元素重叠堆叠**
Agent 在同一页面放置多个图表时，未验证坐标是否冲突，导致图表上下堆叠遮挡。纠正方案：每个图表插入后必须检查其边界（left+width, top+height）是否与已有元素重叠，重叠则调整坐标或缩小尺寸。

**9. 结尾页添加文字内容**
Agent 在结尾页（封尾页）添加总结文字、数据结论等内容。纠正方案：结尾页仅使用模板内置的封尾图片，禁止添加任何文字内容。最后的总结应放在正文页的最后一页。

**10. 每页信息量过少，碎片化叙事**
Agent 将一个可以一页讲完的故事拆成多页，每页只有一两行文字和一个图表，浪费页面空间。纠正方案：一页应尽可能讲全一个完整的故事点（现状+数据+结论），典型汇报正文2-3页即可覆盖全部内容。参考原则：正文页应包含标题+核心观点+支撑数据+图表+结论，缺一不可。

**11. 信息收集时用纯文本提问而非结构化提问工具**
Agent 在信息收集阶段直接输出一段文本消息包含多个问题，用户需要用自然语言回复。纠正方案：每次提问必须使用 AskUserQuestion 工具，提供2-4个选项让用户选择，禁止直接用纯文本消息提问。每轮最多4个问题，用户可通过"Other"选项自由输入。

**12. 跳过 Spec 模式直接开始生成 PPT**
Agent 在信息收集完成后直接开始生成 PPT，跳过了 spec 模式的计划审核环节。纠正方案：信息收集完成后，必须调用 EnterSpecMode 进入 spec 模式，生成包含页面规划、文案、图表、排版方案的完整执行计划，用户审核并 ExitSpecMode 确认后才能开始生成。

| 使用标准模式还是快速模式？ | 模式选择 | 标准模式-逐步引导（推荐） / 快速模式-一次性提供 |

### 12×6网格通用规范

#### 网格定义

- **总列数**：12列
- **总行数**：6行
- **网格边界**：位于页眉页脚横线之间
- **左右页边距**：各留1列（即内容区域为2-11列）
- **上下页边距**：各留半行

#### 视觉黄金原则

1. **视觉流固定**：从上→下，从左→右
   - 标题永远在最上方
   - 解释文字在左，图表在右是最符合阅读习惯的布局

2. **边距留白**
   - 元素不要顶死1列、12列、1行、6行
   - 左右各留1列、上下留半行留白更精致

3. **同页面统一高度/对齐**
   - 并排两个图表必须起止行完全一致
   - 避免一高一低杂乱

4. **饼图使用小技巧**
   - 单张大饼：居中放置，占据中间大面积
   - 多个小饼：均分栏位并排，大小统一

5. **不要碎切割网格**
   - 尽量以整列（3列、4列、6列）为单位放元素
   - 少用1、2列窄小框，页面零散

#### 快速选型口诀

| 场景 | 选用模式 |
|------|---------|
| 一段文字一张图 | 模式1：左右对半布局 |
| 两个数据对比 | 模式2：上下通栏标题+左右双块 |
| 三个论点 | 模式6：三等分三栏 |
| 四个内容 | 模式4：四宫格 |
| 核心总图 | 模式7：上图下文居中 |
| 极简一页 | 模式8：上下分层 |

### 信息密度原则

> **一页讲全一个故事点，不要碎片化拆页。**

典型汇报PPT的页面规划：
- 封面 × 1
- 目录 × 1
- 正文 × 2-4页（每页包含：标题 + 核心观点 + 支撑数据 + 图表 + 结论）
- 结尾 × 1（仅封尾图片）
- 总计 5-7 页

**正文页信息密度标准：**
- 每页必须包含：标题、核心观点（一句话）、支撑数据/要点（3-5个）、图表（如有）、结论
- 禁止"一页只有一段文字+一个图表"的稀疏页面
- 如果一个主题可以分为"现状"和"计划"，应分为两页，而不是拆成四页

**高密度页面布局参考（学习自实际汇报）：**

| 布局模式 | 结构 | 适用场景 |
|---------|------|---------|
| **KPI+图表+成果卡** | 顶部标题栏 / 左侧4个KPI数值卡 / 右侧大图表 / 底部3个成果卡片 | 推广成效、阶段性总结 |
| **目标+行动矩阵** | 顶部标题栏 / 上方3个量化目标 / 下方2×2行动项矩阵 | 计划页、下一步行动 |

> 这两种布局模式通过组合多个圆角矩形、分隔线、数值卡等元素实现高信息密度，每个元素位置精确控制在网格内。

### 阶段2：Spec 模式计划审核

信息收集完成后，**必须调用 EnterSpecMode** 进入 spec 模式，生成完整的执行计划。

**Spec 内容必须包含：**

1. **页面清单**：每页的版式（封面/目录/正文/结尾）、标题、核心观点、支撑要点
2. **排版方案**：每页的12×6网格坐标分配（标注每个元素的起始列-终止列、起始行-终止行）
3. **图表信息**：每页的图表类型、数据来源、网格占位尺寸（宽×高，像素）
4. **信息密度自检**：每页是否满足"标题+观点+数据+图表+结论"
5. **预计总页数**和版式使用清单（如：封面×1, 目录×1, 正文×2, 结尾×1）

**Spec 示例片段：**

```
第3页 - 正文（现状）
- 标题：ECC 部门 Qoder 使用量增长
```

### 阶段3：图表绘制

> **核心原则：先规划网格，再计算尺寸，最后生成图表。不能先按默认尺寸生成图表再强制适配。**

**流程：**

1. **选择图表类型**：根据 Spec 计划中用户确认的图表类型，从35+种图表中选择对应模板

2. **准备数据**：将用户提供的数据整理为图表所需的格式（JSON/CSV）

3. **计算网格占位尺寸（必须先于图表生成）**：
   - 根据 Spec 计划中的12×6网格坐标，计算图表占位区域的实际宽高
   - 计算公式：宽度 = (终止列 - 起始列 + 1) × 列宽，高度 = (终止行 - 起始行 + 1) × 行高
   - 列宽 = (页面安全宽度 - 左右边距) / 12，行高 = (页面安全高度 - 上下边距) / 6
   - 将英寸转换为像素：像素 = 英寸 × 150（dpi）
   - 记录计算结果：每个图表的 `{width_px, height_px, aspect_ratio}`

4. **生成图表HTML**：基于 `全品类图表模板_v1.0.html`，使用Playwright渲染图表
   - 复制对应图表的ECharts配置或CSS/SVG代码
   - 替换为用户的数据
   - **按步骤3计算的像素尺寸设置图表容器尺寸**（禁止使用默认800×600）
   - 确保图表宽高比与网格占位区域一致，避免拉伸变形

5. **渲染为图片**：
   - 使用Playwright/Puppeteer打开生成的HTML
   - 截图保存为PNG/JPG
   - 图片命名：`{页码}_{图表类型}_{主题}.png`

6. **图片质量检查**：
   - 检查图片是否清晰、文字是否可读
   - 检查颜色是否符合品牌规范
   - **检查宽高比是否与网格占位区域匹配（容差 ≤ 5%）**

### 阶段4：生成PPT

**输入：**
1. 模板：`模板.pptx`（封面/目录/标题/正文/结尾版式，作为skill静态资源）
2. 文案：用户确认的每页文案
3. 排版：用户确认的12×6网格坐标
4. 图表：用户确认的图表图片

**生成步骤：**
1. 复制模板对应版式
2. 替换文案（标题、正文等）
3. 按12×6网格坐标放置元素
4. 插入图表图片到指定位置（使用阶段3计算的尺寸，禁止强制拉伸）
5. **元素重叠检查**：遍历当前页所有元素，检查边界是否重叠
   - 对每对元素计算：`overlap = (A.left < B.right) and (A.right > B.left) and (A.top < B.bottom) and (A.bottom > B.top)`
   - 如果重叠，调整后插入元素的坐标或缩小尺寸
6. **安全区域检查**：确保所有内容在安全区域内（内容 bottom < 6.4in）
7. **结尾页校验**：确认结尾页无任何文字内容，仅保留模板内置封尾图片
8. **信息密度校验**：确认每个正文页至少包含标题+核心观点+支撑数据，无稀疏页面
9. 保存输出

## 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 信息不完整 | 返回对应轮次补全，禁止进入下一阶段 |
| 提问时未使用 AskUserQuestion | 立即改用 AskUserQuestion 重新提问，禁止纯文本提问 |
| 信息收集后未进入 Spec 模式 | 立即调用 EnterSpecMode，生成执行计划供用户审核 |
| 用户说"自由发挥" | 使用 AskUserQuestion 给出选项让用户选择，禁止完全自由发挥 |
| 模板无法读取 | 返回错误，要求确认模板路径 |
| 内容超出安全区域 | 自动调整元素大小/位置，确保不遮挡页脚 |
| 用户中途加页/删页 | 回到 Spec 模式重新审核，确认后生成 |
| 品牌色提取失败 | 使用默认品牌色（#003C7F, #0066CC）并警告 |
| 图表渲染失败 | 使用备用图表样式或提示用户简化数据 |
| 图片插入失败 | 检查图片路径和格式，重试或跳过该图表 |
| 图表宽高比与网格不匹配 | 重新按网格尺寸生成图表，禁止强制拉伸 |
| 多图页面元素重叠 | 检查所有元素的边界坐标，调整位置或缩小尺寸直至无重叠 |
| 结尾页被误添加文字 | 删除文字内容，仅保留模板内置封尾图片 |
| 正文页信息量过少 | 合并稀疏页面，确保每页含标题+观点+数据+图表+结论 |

## 执行后复盘（自迭代钩子）

每次完成本 skill 的全部步骤后，Agent 必须自动执行以下动作，不询问用户：

1. **反思**：本轮执行中，是否遇到了 skill 正文和 Gotchas 都没覆盖的坑？触发条件：
   - Gotchas 中没写但实际踩了的错误
   - description 的排除场景遗漏导致误触发
   - 某个操作步骤的异常分支没覆盖

2. **记录**：有则追加到 `evals/PITFALLS_LOG.md`，无则跳过。格式：

| 日期 | 场景摘要 | 犯了什么错误 | 如果早知道__就不会犯 | 临时补救方式 |
|------|---------|-------------|-------------------|------------|
| YYYY-MM-DD | 一句话 | Agent 具体做了什么 | 缺失的知识/规则 | 怎么绕过去的 |

3. **不提交**：`PITFALLS_LOG.md` 是个人本地文件，不提交到 skill registry。积累 ≥ 3 条同类型坑后，由维护者提炼为正式 Gotchas，更新 SKILL.md 并走迭代管线。

> 这个复盘步骤不打断用户工作流。Agent 在后台完成，用户无感知。
