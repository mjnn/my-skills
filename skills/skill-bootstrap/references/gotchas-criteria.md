# Gotchas 提炼标准

> Read this when: 第一步提炼 Gotchas 时

## 什么是 Gotchas

Gotchas 是 Agent **在没有这个 skill 时会犯的具体错误**。每一条 Gotchas 对应一个真实场景中的坑，而不是泛泛的"注意质量"。

## 提炼方法

1. 让作者回忆 3-5 个"Agent 上次做这个任务时犯了什么错"的真实场景
2. 从错误中提炼模式：Agent 做了什么（错误行为）→ 导致什么后果 → 应该怎么纠正
3. 写成结构化的 Gotchas 条目

## 合格标准

每一条 Gotchas 必须同时满足：

| 维度 | 要求 | 反面示例 |
|------|------|---------|
| 具体性 | 描述 Agent 具体会犯什么错误 | ❌ "注意代码质量" → ✅ "Agent 会在没有检查的情况下直接 git add . 提交 venv/"" |
| 纠正方案 | 给出 Agent 应该怎么做的明确指令 | ❌ "要小心" → ✅ "在 git add 之前必须先检查 .gitignore 是否存在" |
| 基于真实 | 来自作者或团队的真实踩坑经验 | ❌ 为了凑数编造 → ✅ "上次我们的项目因为...导致..." |

## 常见陷阱

- **编造型**："记得检查输入" — 太泛，任何场景都适用
- **凑数型**："确保输出正确" — 废话，没有提供任何信息
- **事后型**："注意测试" — 不是 skill 特有的，是通用常识

## 数量要求

- 最少 5 条
- 最好 7-10 条
- 如果作者想不出 5 条，说明对这个场景的理解还不够深入，应该继续挖掘
