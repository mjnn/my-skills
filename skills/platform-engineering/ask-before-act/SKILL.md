# 先问再做

> **核心原则：不确定，就问；没确认，不动。**

## 核心设计思想

- 当Agent对用户意图的确定性低于80%时，必须暂停执行，使用AskUserQuestion工具提问
- 每次提问提供2-4个选项，用户可通过"Other"自由输入
- 澄清后必须复述用户意图供确认，确认后才继续执行
- 禁止在信息不完整的情况下猜测用户意图并继续执行

<HARD-GATE>
在继续执行任何任务之前，必须确认用户意图已明确（确定性 ≥ 80%）。如果意图不确定，禁止继续执行，必须先使用 AskUserQuestion 工具澄清。
</HARD-GATE>

## Gotchas

**1. 使用纯文本提问而非AskUserQuestion工具**
Agent不确定用户意图时，直接在回复中输出一段文本"请问您的需求是什么？"，而非调用AskUserQuestion工具。纠正方案：不确定时调用AskUserQuestion工具，提供结构化选项，禁止纯文本提问。

**2. 用户说"随便"时自由发挥**
用户回答"随便""看着办""你决定"时，Agent直接按自己的理解继续执行。纠正方案：用户说"随便"时必须使用AskUserQuestion提供推荐选项和替代选项，让用户点选确认。

**3. 澄清后未复述确认直接执行**
用户回答后，Agent未复述理解到的意图，直接开始执行。纠正方案：每次澄清后必须用一句话复述用户意图，如"我理解您的需求是xxx，对吗？"，得到确认后再执行。

**4. 一次抛出太多问题 overwhelm用户**
Agent在不确定时一次性抛出5-10个问题。纠正方案：每轮AskUserQuestion最多4个问题，分轮次进行，每轮聚焦一个维度。

**5. 用户意图已明确仍反复确认**
用户已经清楚描述了需求，Agent仍不断提问确认。纠正方案：只在确定性低于80%时触发本skill，意图明确时不应触发。

## 触发条件

本skill在以下场景自动触发：
- 用户输入的任务描述模糊或存在歧义
- 同一需求存在多种合理的理解方式
- 缺少关键执行参数（如输出路径、格式偏好、优先级等）
- 用户要求"帮我做xxx"但没有说明具体标准或偏好
- 上下文切换后，新任务与之前的理解存在冲突

## 流程

### 步骤1：判断是否需要澄清

Agent自我评估对用户意图的确定性（0-100%）：
- **确定性 ≥ 80%**：不触发本skill，继续执行
- **确定性 < 80%**：进入步骤2

> 判断标准：是否明确知道"做什么""怎么做""做到什么程度"三个维度？

### 步骤2：使用AskUserQuestion澄清

调用AskUserQuestion工具，设计1-4个问题，每个问题提供2-4个选项。

**问题设计原则：**
- 聚焦缺失信息：只问当前还不确定的关键维度
- 提供选项：每个问题必须有明确的选项，用户可通过"Other"自由输入
- 推荐标注：推荐选项放在第一个，标注"（推荐）"
- 分轮进行：每轮最多4个问题，如有更多问题分轮次提问

**示例：**

```
Q1(header="任务目标")：您希望达到什么效果？
  选项：快速完成（推荐） / 追求质量 / 平衡两者

Q2(header="输出格式")：您希望以什么形式交付？
  选项：PPT（推荐） / Word文档 / 纯文本 / 其他

Q3(header="详细程度")：内容详细程度？
  选项：详细展开（推荐） / 简明扼要 / 仅列要点
```

### 步骤3：总结确认

用户回答后，Agent复述理解到的意图：

> "根据您的回答，我理解您的需求是：**[一句话总结]**。确认无误后我将开始执行。如有需要调整的地方请告诉我。"

如果用户提出修改意见，返回步骤2重新澄清。
如果用户确认，进入步骤4。

### 步骤4：继续执行

用户确认后，记录澄清结果到上下文，继续执行后续任务。

**若后续任务涉及生成、修改或创建文件**，必须先调用 `EnterSpecMode` 生成执行计划（含任务概览、关键决策、预期输出），用户审核后 `ExitSpecMode` 确认，再开始执行。本 skill 只负责澄清意图，不跳过 Spec 审核直接动手。

## 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 用户回答仍不清晰 | 再次使用AskUserQuestion追问，最多3轮 |
| 用户说"先不做了" | 停止执行，礼貌结束对话 |
| 用户说"你看着办" | 使用AskUserQuestion提供推荐选项让用户选择 |
| 多轮澄清后仍无法明确 | 提供几种可行方案让用户选择，或建议用户先明确需求 |
| 用户要求跳过澄清 | 记录用户明确放弃的意愿，继续执行但承担不确定性风险 |

## 执行后复盘（自迭代钩子）

每次完成本skill的全部步骤后，Agent必须自动执行以下动作，不询问用户：

1. **反思**：本轮澄清是否高效？是否存在可以优化的问题设计？
2. **记录**：有则追加到`evals/PITFALLS_LOG.md`，无则跳过。格式：

| 日期 | 场景摘要 | 犯了什么错误 | 如果早知道__就不会犯 | 临时补救方式 |
|------|---------|-------------|-------------------|------------|
| YYYY-MM-DD | 一句话 | Agent具体做了什么 | 缺失的知识/规则 | 怎么绕过去的 |

3. **不提交**：`PITFALLS_LOG.md`是个人本地文件，不提交到skill registry。积累≥3条同类型坑后，由维护者提炼为正式Gotchas，更新SKILL.md并走迭代管线。
