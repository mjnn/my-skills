# 子 Agent 编排执行

按计划逐任务派子 Agent 执行，每项任务完成后做审查（方案符合度 + 质量），审查通过才进入下一任务。

<HARD-GATE>
1. 必须有完整的实施计划才能开始执行，禁止无计划派子 Agent。
2. 每个任务完成后必须通过两阶段审查（方案符合度 + 质量），禁止跳过审查。
3. 审查发现的问题必须修复后重新审查，禁止"先这样"继续推进。
4. 禁止同时派多个实现子 Agent，必须串行执行。
</HARD-GATE>

**核心理念：** 每个任务派一个全新上下文的子 Agent + 完成后审查 = 高质量、快迭代

**持续执行：** 不要在任务之间停下来问用户"要继续吗？"。一口气执行完所有任务。唯一停下来的理由：遇到无法解决的阻塞、真正的模糊不清、或者全部任务已完成。

## 能力边界

### 本技能负责
- 按已确认的实施计划逐任务派出子 Agent 执行
- 每项任务完成后的审查（方案符合度 + 质量）
- 适用于任务间相对独立的场景

### 本技能不负责
- 任务规划 → 使用 **writing-plans**
- 需求梳理和设计 → 使用 **requirement-briefing**
- 最终交付验证 → 使用 **verification-before-completion**
- 任务高度耦合（后面依赖前面中间产物）的场景
- 单任务即可完成的简单工作

## 执行流程



### 运行时说明（无脚本）

本 skill 以对话与流程引导为主，不直接执行脚本。若会话中需要运行**本 skill 或其他 skill 的 scripts/**，必须先确认 skill-kit 已就绪（Read skill-kit `references/preflight.md`）；未就绪则引导用户运行 **skill-env-setup**。

开始执行前，调用 EnterSpecMode 进入 spec 模式，展示任务列表和执行策略。用户审核后调用 ExitSpecMode 确认，才开始逐任务执行。

```
读计划 → 提取所有任务 → 创建 TodoWrite
    ↓
┌─ 对每个任务循环 ─────────────────────┐
│  1. 派实现子 Agent（提供完整任务文本+上下文）│
│  2. 子 Agent 有疑问？→ 回答后重新派      │
│  3. 子 Agent 完成并自检                 │
│  4. 审查：方案符合度检查                 │
│  5. 审查：质量检查                      │
│  6. 有问题？→ 子 Agent 修 → 回到步骤 4   │
│  7. 标记任务完成                       │
└──────────────────────────────────────┘
    ↓
全部完成后 → 收尾验证
```

## 派子 Agent 的要点

**必须提供：**
- 完整任务文本（不要让子 Agent 自己去读计划文件）
- 任务在整个计划中的位置和上下文
- 涉及的文件路径和预期产出
- 明确的成功标准

**模型选择：**
- 机械执行类任务（单文件、清晰指令）→ 用轻量模型
- 需要判断协调的任务（多文件、模式匹配）→ 用标准模型
- 需要设计判断或全局理解的任务 → 用最强模型

## 子 Agent 返回状态处理

- **DONE** — 进入审查流程
- **DONE_WITH_CONCERNS** — 完成了但自己标注了疑虑。先读疑虑再决定是否回修
- **NEEDS_CONTEXT** — 缺信息。补充后重新派
- **BLOCKED** — 卡住了。使用 AskUserQuestion 工具向用户说明情况并提供选项：补充上下文 / 升级模型 / 拆小任务 / 暂停。评估后重新派或暂停。

绝不忽视子 Agent 的升级信号。

## 审查流程

每项任务完成后，做两阶段审查：

**1. 方案符合度审查：**
- 产出是否覆盖了计划中的所有要求？
- 有没有多做计划外的东西？（多做也是问题）
- 产出格式、命名、路径是否符合计划？

**2. 质量审查：**
- 逻辑是否正确？
- 有没有明显的错误或遗漏？
- 文件组织是否合理？

只有两项都通过，才标记任务完成并进入下一任务。

## Gotchas

1. **跳过审查直接进下一任务** — 审查是质量防线，不能跳过。Agent 常在时间压力下跳过审查，导致问题累积。
2. **同时派多个实现子 Agent** — 会冲突，必须串行执行。Agent 常为了效率同时派发多个任务，导致文件冲突。
3. **让子 Agent 自己读计划文件** — 必须提供完整任务文本。Agent 常偷懒让子 Agent 去读文件，导致上下文丢失。
4. **子 Agent 返回 DONE_WITH_CONCERNS 时忽视疑虑** — 必须先处理疑虑。Agent 常急于推进而忽略子 Agent 的警告信号。
5. **审查发现问题不修就继续** — 必须修完再查。Agent 常在审查发现小问题后说"先这样"继续推进。

## 红线

**绝对不要：**
- 跳过审查（方案符合度或质量一个都不能少）
- 审查发现问题但不修就继续
- 同时派多个实现子 Agent（会冲突）
- 让子 Agent 自己去读计划文件（你必须提供完整文本）
- 不提供场景上下文
- 忽略子 Agent 的问题
- "差不多就行"
- 跳过复审查（修改后必须再查）
- 方案审查没通过就开始质量审查（顺序不能乱）
- 审查有未解决问题就进下一任务

## 执行后复盘（自迭代钩子）

每次完成本 skill 的全部步骤后，Agent 必须自动执行以下动作，不询问用户：

1. **反思**：本轮执行中，是否遇到了正文和 Gotchas 都没覆盖的坑？触发条件：
   - Gotchas 中没写但实际踩了的编排问题
   - 某种任务类型的子 Agent 派发策略不当
   - 审查流程遗漏了某个维度

2. **记录**：有则追加到 `evals/PITFALLS_LOG.md`，无则跳过。格式：

| 日期 | 场景摘要 | 犯了什么错误 | 如果早知道__就不会犯 | 临时补救方式 |
|------|---------|-------------|-------------------|------------|
| YYYY-MM-DD | 一句话 | Agent 具体做了什么 | 缺失的知识/规则 | 怎么绕过去的 |

3. **不提交**：`PITFALLS_LOG.md` 是个人本地文件，不提交到 skill registry。积累 >= 3 条同类型坑后，由维护者提炼为正式 Gotchas，更新 SKILL.md 并走迭代管线。
