# 车端特有要素与设计原则

> Read when: 生成事件参数或校验车端场景覆盖时

## 屏幕编号规范

| 编码 | 屏幕 |
|------|------|
| 0 | 主驾屏（中控） |
| 1 | 副驾屏 |
| 2 | 二排左门屏 |
| 3 | 二排右门屏 |
| 4 | HUD |
| 5 | 后排中央屏 |

## 触发方式编码

| 编码 | 触发方式 |
|------|----------|
| 0 | 触屏点击 |
| 1 | 触屏滑动 |
| 2 | 语音（SDS） |
| 3 | 物理按键 |
| 4 | 手势 |
| 5 | 远程触发（手机APP/云端） |

## 数据分类标准

| 分类 | 适用范围 |
|------|----------|
| 车辆使用数据 | 用户与车辆交互的行为数据 |
| 车辆控制数据 | 车辆硬件控制相关数据 |
| 配置标定数据 | 系统设置、偏好配置类数据 |
| 故障诊断数据 | 错误、异常、诊断类数据 |
| 行驶数据 | 导航、驾驶辅助相关数据 |
| 媒体娱乐数据 | 音乐、视频、电台等数据 |

## 设计原则

1. **隐私合规**：涉及用户个人信息的参数必须标注脱敏要求
2. **事件唯一性**：事件ID 全局唯一，不可重复
3. **属性一致性**：同一业务含义的属性名在不同事件中保持一致
4. **屏幕区分**：车端多屏场景必须记录 `screen` 属性
5. **触发方式区分**：不同交互方式（触屏/语音/按键）必须记录 `trigger_method`
6. **覆盖完整性**：覆盖正常流程、异常流程、边界场景
7. **分析可量化**：每个事件都要有明确的分析目的和可量化的核心指标
