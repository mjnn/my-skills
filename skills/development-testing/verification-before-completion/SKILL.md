# 完成前验证

"做完了"三个字说出来之前，必须验证它真的做完了。这是最后一道防线。

<HARD-GATE>
1. 在向用户报告"做完了"之前，必须完成全部 5 项验证清单（产出存在性、完整性、逻辑验证、计划对照、清理收尾）。
2. 验证发现的问题必须修复后从头重新验证，禁止"先这样"跳过。
3. 用户明确说"不用验证"时，可以跳过。
</HARD-GATE>

## 能力边界

### 本技能负责
- 任务完成后的最终验证（产出存在性、完整性、逻辑、计划对照、清理）
- 防止"看起来好了"就交差
- 作为交付前的最后一道防线

### 本技能不负责
- 任务执行过程中的质量把控 → 由执行型 skill 自行保证
- 需求梳理 → 使用 **requirement-briefing**
- 任务规划 → 使用 **writing-plans**
- 用户明确说"不用验证"时的强制验证

## Gotchas

1. **"应该没问题吧"** — 去检查，不要凭感觉。Agent 常在时间压力下跳过验证，导致"差不多"就是差很多。
2. **只验证产出存在性，不验证内容正确性** — 必须打开文件检查内容。Agent 常只看文件存在就标记完成。
3. **发现问题后只修表面，不复查** — 修完必须从头验证。Agent 常修完一处就不验证其他部分。
4. **忽略"清理收尾"步骤** — 临时文件和乱掉的工作区会留下技术债。Agent 常忘记清理验证过程中产生的临时文件。
5. **用户催就跳过验证** — 验证比速度更重要。Agent 常在用户催促下说"做完了"，但实际未验证。

## 何时触发

- 完成一个独立任务后
- 完成一个实施计划的所有任务后
- 用户说"好了吗"而你想说"好了"的时候

## 验证清单

按顺序过：

### 1. 产出存在性
- 计划中要求的每个产出文件，真的存在吗？
- 路径正确吗？命名正确吗？
- 不是空的吧？

### 2. 产出完整性
- 打开每个产出文件，内容符合预期吗？
- 有没有截断、乱码、格式错误？

### 3. 逻辑验证
- 结果在逻辑上说得通吗？
- 有没有明显反常的结论？
- 用常识做一次快速 sanity check

### 4. 计划对照
- 逐条对照计划中的成功标准
- 每一项都满足了吗？
- 有没有遗漏的子任务？

### 5. 清理收尾
- 有没有临时文件需要清理？
- 工作区有没有因为这次任务变乱？
- 产出文件放对位置了吗？

## 发现问题时

- **小问题**（格式、命名、路径）→ 直接修掉，再从头验证
- **中等问题**（内容正确但不符合计划）→ 修掉，再从头验证
- **大问题**（逻辑有误、缺关键产出）→ 退回执行阶段，完成后重新验证
- **无法判断的问题**→ 诚实告诉用户，说明验证了什么、发现了什么、你的判断。使用 AskUserQuestion 工具提供选项：继续交付 / 退回修复 / 暂停等待更多信息。

## 防自欺规则

这些想法出现时立刻停下来：

| 想法 | 现实 |
|------|------|
| "应该没问题吧" | 去检查 |
| "我之前看过了" | 再看一次，现在状态可能变了 |
| "这个不太重要" | 用户觉得重要才写进计划的 |
| "差不多了" | 差不多就是差很多 |
| "用户自己会检查的" | 你是最后一道防线 |

## 验证通过后

向用户报告时，用这个格式：

```
✅ [任务名] 已完成

产出：
- `path/to/output1`：[简述]
- `path/to/output2`：[简述]

验证：产出存在性 ✅ / 完整性 ✅ / 逻辑 ✅ / 计划对照 ✅
```

## 执行后复盘（自迭代钩子）

每次完成本 skill 的全部步骤后，Agent 必须自动执行以下动作，不询问用户：

1. **反思**：本轮验证中，是否有正文和 Gotchas 都没覆盖的坑？触发条件：
   - Gotchas 中没写但实际踩了的验证盲区
   - 某类产出文件的验证方式遗漏
   - 验证清单的某项不够具体

2. **记录**：有则追加到 `evals/PITFALLS_LOG.md`，无则跳过。格式：

| 日期 | 场景摘要 | 犯了什么错误 | 如果早知道__就不会犯 | 临时补救方式 |
|------|---------|-------------|-------------------|------------|
| YYYY-MM-DD | 一句话 | Agent 具体做了什么 | 缺失的知识/规则 | 怎么绕过去的 |

3. **不提交**：`PITFALLS_LOG.md` 是个人本地文件，不提交到 skill registry。积累 >= 3 条同类型坑后，由维护者提炼为正式 Gotchas，更新 SKILL.md 并走迭代管线。
